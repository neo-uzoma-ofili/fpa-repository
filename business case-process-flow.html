Business Case Skill — Process Flow

Full 8-step workflow for business case analysis: data sources, analytical tools, confirm gates, and outputs at each stage.

Skill: /liftoff
Version: v1.4.0
Data: analytics.earl (Snowflake)
Analysis: Hex + Claude
Output: Google Doc
ℹ
How to read this diagram Each step is confirmation-gated — Claude does not advance until the user explicitly confirms. CONFIRM GATE markers show these checkpoints. Hex steps trigger Snowflake queries via Hex threads. Pills show inputs input, outputs output, data tables data, and tools called tool.
1
Initiative Intake & Interpretation
Claude
Read the business case from Google Doc, Confluence, or a written description. Summarize the initiative, identify the target program and unit economic levers, then classify as Frontbook, Backbook, or Both using the Company Funnel Lever Taxonomy.
Inputs
Google Doc URL Confluence page URL Written description
Outputs
Program classification (Banking / Lending / Spend / Growth) Unit economic levers (primary + secondary) Frontbook / Backbook / Both classification Key assumptions + missing info
CONFIRM GATE — classification & lever mapping
2
Eligible Population Definition
Hex analytics.earl
Define who qualifies for the initiative. Route to one of three paths based on the Step 1 classification, confirm the population definition with the user, then trigger Hex thread(s) to size the eligible population by credit band.
Data Sources
earl_account (daily grain — filter to latest date) earl_customer earl_application
Routing — driven by Step 1 Frontbook / Backbook / Both classification
Path A
Frontbook
  • Funnel lever lookup table determines eligible set
  • Denominator: profiles created (Volume) or total customers (Conversion / Activation)
  • Single Hex thread — group by originalcreditscoreband
Path B
Backbook
  • All existing customers, active + inactive
  • Denominator: total customer base — do NOT filter to active-only
  • Single Hex thread — group by originalcreditscoreband
Path C
Both — Parallel
  • Confirm both definitions before triggering
  • Track A (Frontbook) + Track B (Backbook) definitions presented together
  • 2 Hex threads triggered in parallel
Outputs
Population table — N by credit band % of eligible vs. total denominator Hex thread link(s)
CONFIRM GATE — population definition & sizing
3
Population Segmentation
Claude
Break the eligible population into sub-groups with meaningfully different adoption rates. Three layers — Layer 1 is always applied. Layers 2 and 3 are conditional on program and lever type; rationale required if either is skipped.
Layer 1 — Always Applied
Credit Band
6 bands: Sub Prime → Near-Prime → Prime → Prime+ → Super Prime → None (money-only). Compulsory on every initiative.
Layer 2 — Growth: Mandatory · Others: Optional
Acquisition Channel
Activations · Paid · Organic · C2C · Affiliate · Other. Must state rationale if not applied.
Layer 3 — Reactivation: Mandatory · Others: Optional
Activity State
Highly Active (<30d) · Moderately Active (31–90d) · Low/Dormant (90d+). Must state rationale if not applied.
Outputs
Segment matrix definition (N segments) Layer 2 / Layer 3 rationale documented
CONFIRM GATE — segmentation approach
4
Baseline Behaviours
Hex analytics.earl
Establish the pre-initiative diagnostic profile of each segment. Metric set is program-specific and driven by the Step 1 lever mapping. Analysis period: trailing 3 complete calendar months to the most recent available month.
Data Sources
earl_account (latest date — balance & product metrics) earl_transaction (trailing 3-month spend metrics)
Program-Specific Metric Sets
Banking Avg deposit balance · Monthly attrition rate · Product activation rate · Reactivation rate
Lending Avg credit balance & utilization · GCO rate (ann.) · Bucket roll rates · Recovery rate · Avg credit limit & ICL
Spend Avg monthly purchase volume · Activity rate · Effective rewards rate · Transaction margin per active
Growth Monthly new acquisitions · Channel mix (Marketing taxonomy) · C2C rate
Outputs
Baseline table — segments × program metrics Key observations Hex thread link
CONFIRM GATE — baselines reasonable?
5
Adoption Likelihood
Claude
Estimate the probability each segment adopts the initiative. Produces Base / Downside / Upside scenarios anchored to observable behaviour or documented business case assumptions. Credit band alone is not a reliable predictor — rates are initiative-specific. Surface program driver prompts before building the adoption table.
Inputs
Business case adoption assumptions Step 4 baseline metrics (for differentiation) Program driver prompts (Banking / Lending / Spend / Growth)
Outputs
Adoption table — Downside / Base / Upside by segment Adopted N (Base) by credit band Assumption log (updated) Historical anchors
CONFIRM GATE — adoption assumptions reasonable?
6
Impact Estimation
Claude
Apply unit economics to the adopted population. For Both-path initiatives, produces separate Frontbook and Backbook sections plus a blended summary table. Flags items requiring FP&A 3-statement model build. If unit economics are missing from the business case, Claude proposes estimates from Step 4 baselines (clearly labelled as estimated).
Inputs
Business case unit economics Step 4 baselines (fallback for missing unit econ) Step 5 adopted N — by scenario
Outputs
Impact table — segment × scenario Monthly + annualized impact (Down / Base / Up) Both-path: Frontbook + Backbook + blended table FP&A model handoff items
CONFIRM GATE — impact assumptions confirmed?
7
Timeline
Claude
Extract the estimated launch date from the business case to anchor the FP&A model timeline. If not stated, Claude prompts the user directly before proceeding. No Hex query required.
Input
Business case (launch date field)
Output
Estimated launch: YYYY-MM
CONFIRM GATE — timeline confirmed
8
Final Output Document
Google Doc Claude
Copy the output doc template to the Claude Drive folder, populate all {{PLACEHOLDER}} tokens via find_and_replace_doc, and share the link. The document contains an 8-section structure covering Executive Summary through References, with an embedded auditability checklist.
Tools Called
copy_drive_file — template ID 166Jq9H… → Claude folder find_and_replace_doc — one call per {{PLACEHOLDER}} token
Output Document — 8 Sections
1. Executive Summary 2. Problem Statement 3. Methodology 4.1 Eligible Population 4.2 Segmentation 4.3 Baseline Behaviours 4.4 Adoption Likelihood 4.5 Impact Estimation 5. Program Lever Mapping 6. Timeline 7. Conclusion & FP&A Handoff 8. References
